# f-27: Fast Reinstall — Skip PM When Package Dependencies Unchanged

- **Milestone:** M1
- **Complexity:** Low
- **Depends On:** #019 (strip devDependencies)

## 1. Problem Statement

`kley install <pkg>` always delegates to the native package manager (e.g. `npm install ./.kley/my-lib`), even when reinstalling a package whose `dependencies` and `peerDependencies` haven't changed since the last install. The PM call is the slowest part of the operation and is unnecessary in this case.

This is especially common during iterative development: a developer runs `kley install my-lib` after each `kley publish` in the library, but the library's dependency list rarely changes between iterations — only the source code does.

## 2. Proposed Solution (v2)

The `kley install` command will be optimized to skip calling the package manager (PM) when a package's dependencies have not changed. This is achieved by introducing a "Fast Path" that intelligently checks the state of `node_modules`.

### High-Level Logic

1.  **Preparation**: Copy the package from the kley store to the project's local cache at `./.kley/<pkg>`.
2.  **Dependency Check**: Compare the package's `dependencies` and `peerDependencies` against the snapshot stored in `kley.lock`.
    *   If dependencies have changed or no snapshot exists, fall back to the **Slow Path** (run the PM).
    *   If dependencies are unchanged, proceed to the **Fast Path**.

### Fast Path Logic

When the Fast Path is triggered, `kley` will inspect the destination directory at `node_modules/<pkg>` and act as follows:

*   **Case A: Destination is a symlink to kley's cache.**
    *   **Condition**: `node_modules/<pkg>` is a symlink pointing to `./.kley/<pkg>`.
    *   **Reasoning**: This state can be created by a previous `kley install` (with a modern PM) or a `kley link`. The link is already correct and the source files in the cache have just been updated.
    *   **Action**: Do nothing. The installation is already up-to-date. Report success.

*   **Case B: Destination is an unknown symlink.**
    *   **Condition**: `node_modules/<pkg>` is a symlink pointing to any location *other than* `./.kley/<pkg>`.
    *   **Reasoning**: This is an unexpected state, likely created by `npm link` or manual user intervention. The safest action is to let the project's configured PM handle it.
    *   **Action**: Fall back to the **Slow Path**.

*   **Case C: Destination is a regular directory.**
    *   **Condition**: `node_modules/<pkg>` is a regular directory (not a symlink).
    *   **Reasoning**: This is the standard scenario after a previous `kley install`. Files are already registered in `node_modules` via a prior PM call; only the source code changed.
    *   **Action**: Copy the package contents from `./.kley/<pkg>` directly into `node_modules/<pkg>`, overwriting any existing files. Report success.

*   **Case D: Destination does not exist.**
    *   **Condition**: `node_modules/<pkg>` is absent.
    *   **Reasoning**: This is a fresh install scenario (new machine, after `rm -rf node_modules`, etc.). The package manager must be invoked so it registers the `file:` dependency in `package.json` and wires up `node_modules` properly.
    *   **Action**: Fall back to the **Slow Path**.

### Slow Path Logic

The Slow Path remains unchanged:
1.  Run the native package manager (e.g., `npm install ./.kley/<pkg>`).
2.  After the PM succeeds, update the dependency snapshot for the package in `kley.lock`.

### Dependency Snapshot in `kley.lock`

This remains as originally specified. A snapshot of `dependencies` and `peerDependencies` will be stored in `kley.lock` after each successful Slow Path installation.

```rust
pub struct PackageInfo {
    pub version: String,
    /// Snapshot of dependencies at last PM install, used for fast reinstall check
    #[serde(default, skip_serializing_if = "BTreeMap::is_empty")]
    pub dependencies: BTreeMap<String, String>,
    /// Snapshot of peer dependencies at last PM install, used for fast reinstall check
    #[serde(default, skip_serializing_if = "BTreeMap::is_empty")]
    pub peer_dependencies: BTreeMap<String, String>,
}
```

## 3. When Fast Path Applies

| Scenario | Snapshot exists? | Dependencies changed? | `node_modules/<pkg>` state | Path |
|----------|-----------------|----------------------|----------------------------|------|
| First `kley install` | No | N/A | Any | **Slow** — run PM |
| Reinstall after source change | Yes | No | Directory exists | **Fast** — copy to node_modules |
| Reinstall after source change | Yes | No | Symlink to `.kley/<pkg>` | **Fast** — do nothing |
| Reinstall after source change | Yes | No | Does not exist | **Slow** — run PM (registers `file:` dep) |
| Reinstall after source change | Yes | No | Unknown symlink | **Slow** — run PM |
| Reinstall after new dep added | Yes | Yes | Any | **Slow** — run PM |
| `kley add` (first time) | No | N/A | Any | **Slow** — run PM |
| `kley link` | N/A | N/A | N/A | **N/A** — symlink, no PM |

## 4. Acceptance Criteria

- When `kley install <pkg>` is called and the package's `dependencies` and `peerDependencies` match the snapshot in `kley.lock`, the PM call is skipped and files are copied directly to `node_modules/<pkg>/`.
- When `dependencies` or `peerDependencies` have changed, the PM is called as usual.
- `kley.lock` stores a snapshot of `dependencies` and `peerDependencies` after each PM install.
- First-time install (no snapshot in `kley.lock`) always runs the PM.
- `kley link` is unaffected (uses symlinks, no PM call).
- The optimization works with all package managers (npm, pnpm, yarn) — comparison is JSON-only, no PM-specific logic.
