# f-35: Skip PM When Package Dependencies Already Satisfied

- **Milestone:** M2
- **Complexity:** High
- **Depends On:** f-26 (skip PM for dependency-less packages)
- **Status:** Needs research — this ticket captures the problem and constraints, not a committed design.

## 1. Problem Statement

`kley install <pkg>` for a package that **has** dependencies always delegates to the native package manager so it can resolve and install that dependency graph. But in many projects those dependencies are already present in the host `node_modules/` at compatible versions — the library being installed locally usually shares the consuming project's stack. In that case the PM call resolves to a no-op but still costs the full PM startup and resolution time.

f-26 already covers the trivial case (a package with **no** dependencies). This ticket covers the harder case: the package has dependencies, but they are **already satisfied** in the project, so the PM could be skipped safely.

## 2. Why This Needs Research

The original f-26 design proposed this via `npm list --json` parsing and semver matching.
That approach is fragile and incomplete. Before committing to an implementation, the following open problems must be solved:

1. **Source of truth must be `node_modules`, not `package.json`.** The project's
   `package.json` only *declares* dependencies — it does not prove they are installed (the project may have a stale or absent `node_modules`). The actual installed version must be read from `node_modules/<dep>/package.json`.
2. **Semver range matching.** A package declares `"lodash": "^4.0.0"`; the project has `lodash@4.17.21` installed. Deciding that the range is satisfied requires a semver implementation (a crate, or a carefully scoped subset). Exact-string matching avoids the dependency but almost never triggers, so it delivers little.
3. **Transitive and peer dependencies.** "All direct dependencies present" does not imply "fully resolvable" — transitive deps, optional deps, and peer deps can still be missing.
   The check must define how deep it goes and what it assumes.
4. **`npm list --json` fragility.** If that route is taken, its output format varies across PM versions and PMs (npm/pnpm/yarn differ). A PM-agnostic approach (reading `node_modules` directly) may be more robust — to be evaluated.

## 3. Acceptance Criteria

Deferred until the research lands. To be defined once a safe, PM-agnostic detection strategy is chosen. At minimum, any accepted design must:

- never skip the PM when a required dependency is missing or version-incompatible(correctness over speed — a wrong skip leaves the project broken);
- fall back to the PM whenever it cannot prove all dependencies are satisfied;
- behave identically across npm, pnpm, and yarn, or explicitly scope which PMs it supports.
