# f-26: Skip PM for Dependency-less Packages on Install

- **Milestone:** M1
- **Complexity:** Low
- **Depends On:** #017 (install command), #019 (strip devDependencies)
- **Related:** f-27 (fast reinstall), f-32 (link rework / `connection` field), f-35 (deps already satisfied)

## 1. Problem Statement

On a first `kley install <pkg>` (or whenever `node_modules/<pkg>` is absent), the command
always delegates to the native package manager — even for packages that have **no dependencies to resolve**. In that case the PM does nothing useful beyond two steps that kley already performs without a PM:

1. writing the `file:.kley/<pkg>` entry into `package.json`, and
2. placing the package under `node_modules/<pkg>`.

The PM call is the slowest part of the operation, so paying for it when there is no dependency graph to resolve is pure overhead.

> **Note:** the original scope of this ticket (verify a package's dependencies are already satisfied in the host `node_modules` via `npm list --json` / semver matching, behind a `--fast` flag) has been split out. Most of its value was already delivered by **f-27** (skip PM when dependencies are *unchanged* between installs). The harder "deps already satisfied in the project" case is tracked separately in **f-35**.

## 2. Proposed Solution

When the installed package's `dependencies` **and** `peerDependencies` are both empty (or absent), take a direct route instead of calling `pm_install_command`:

1. **Update `package.json`** — unless `--no-save` is set, add `file:.kley/<pkg>` to `dependencies` (or `devDependencies` with `-D`). Reuse the existing
   `add::update_package_json` logic so indentation and existing entries are preserved.
2. **Create a symlink** `node_modules/<pkg>` → `.kley/<pkg>` (not a copy). Reuse the cross-platform symlink helper from `link` (`symlink_dir` with a `create_junction` fallback on Windows).
3. **Skip the PM.**

### Why a symlink, not a copy

- **Matches the PM.** npm 5+ creates a symlink for a `file:` dependency, so a symlink is the faithful outcome — a copy would diverge from what the project's PM would produce.
- **Lands in f-27's steady state.** f-27 Case A already treats "`node_modules/<pkg>` is a symlink to `.kley/<pkg>`" as the canonical up-to-date install state and does nothing on subsequent installs. So this path makes the next `kley install` a clean no-op.
- **Instant `publish --push`.** Because the link points at `.kley/<pkg>`, a later
  `kley publish --push` updates the cache and the change is immediately visible through the symlink — no reinstall needed.
- **Windows is already handled.** `link` falls back to a junction (`mklink /J`) when
  directory symlinks are unavailable; the shared helper carries that over.

### Relationship to `connection` (f-32)

This path sets `connection: "install"` in `kley.lock` and targets `.kley/<pkg>`, keeping it firmly in install semantics. This is distinct from f-32's reworked `link`, which symlinks to the live **source** directory with `connection: "link"`. The two never collide — they differ by both target and connection type. The dependency on f-32 is soft (only the `connection` field); the symlink-to-`.kley` behavior is independent and is in fact the logic f-32 moves `link` away from.

### What changes

| Aspect | `install <pkg>` (with deps) | `install <pkg>` (no deps) |
|--------|-----------------------------|---------------------------|
| Copies to `.kley/<pkg>/` | ✅ | ✅ |
| `kley.lock` updated | ✅ | ✅ |
| `package.json` modified | ✅ (via PM) | ✅ (direct, unless `--no-save`) |
| PM runs | ✅ | ❌ |
| `node_modules/<pkg>` | PM's choice (symlink on npm 5+) | symlink → `.kley/<pkg>` |

## 3. Acceptance Criteria

- A package with no `dependencies`/`peerDependencies` installs **without** invoking the package manager (verified via `pm.log` being absent / unchanged).
- `package.json` gets the `file:.kley/<pkg>` entry — in `dependencies`, or in
  `devDependencies` with `-D`. With `--no-save`, `package.json` is not modified.
- `node_modules/<pkg>` is a symlink pointing to `.kley/<pkg>` (a junction on Windows).
- `kley.lock` is updated as usual, with `connection: "install"`.
- A package **with** dependencies still goes through the package manager (unchanged behavior — regression guard).
- Works across npm, pnpm, and yarn — the decision is based on the package's own `package.json`, with no PM-specific logic.
